The Stack

Four layers. Clear contracts. Parallel progress.

Sauna is a four-layer system. Each layer has a distinct job, a clear interface with its neighbors, and an independent rate of change. Understanding the stack means understanding where work happens, who owns it, and how the pieces connect.

Layer 4 — The Foundation

Infrastructure

Everything the system stands on. Runtime environments, data storage, networking, compute scaling, deployment pipelines, monitoring. This layer should be invisible — if you're thinking about infrastructure, something is wrong.

Owns: Uptime, latency, data integrity, scaling, security fundamentals.

Success looks like: Nobody talks about it. It just works. When intelligence or capabilities need more — more compute, more storage, lower latency — infrastructure delivers without becoming a bottleneck.

Interface: Infrastructure ↔ Capabilities

Capabilities need reliable primitives: "store this," "fetch that," "run this code," "call this API." Infrastructure provides them as stable contracts. The less capabilities needs to think about the underlying infrastructure, the better.

Layer 3 — The Harness

Capabilities

Where inference meets the real world. Tool execution, API integrations, sandbox environments, authentication flows, retry logic, rate limiting, error handling. Capabilities is the harness that lets intelligence act — not just think.

Owns: Tool reliability, integration breadth, execution safety, auth management, the 2000+ integrations.

Success looks like: When intelligence decides to use a tool, it works. Every time. With predictable latency, clear error messages, and graceful degradation. The agent never needs to wonder "is this tool available?" — it just calls it.

Interface: Capabilities ↔ Intelligence — THE CRITICAL INTERFACE

This is where most failures happen. Intelligence decides what to do; capabilities determines whether it can be done. The contract: intelligence gets a stable, well-documented surface of tools and actions. It knows what's available, what each tool does, what inputs it needs, what outputs it returns. Capabilities guarantees execution. Intelligence guarantees good decisions about what to execute.

When this interface is clean, both sides can iterate independently. When it's messy — unclear tool descriptions, unreliable execution, surprising failures — intelligence can't do its job because it can't trust the platform underneath.

Layer 2 — The Brain

Intelligence

How tokens become outcomes. System prompt architecture, reasoning quality, decision logic, context engineering, skills, workflows, evals. This is the layer that decides what to do, when to do it, and how to do it well.

Owns: Decision quality, prompt architecture, context management, skill design, eval frameworks, reasoning fidelity.

Success looks like: The agent makes decisions you'd agree with. It uses the right tools at the right time. It asks when it should ask and acts when it should act. When you review its work, your reaction is "yes, that's what I would have done" — or better, "I wouldn't have thought of that, but it's right."

Intelligence is invisible to the user. They don't see the system prompt, the reasoning chain, the context curation. They see the outcome. Our job is to make the invisible machinery so good that the visible output feels effortless.

Interface: Intelligence ↔ UX/Product

Product shapes what the user sees. Intelligence determines the quality of what they see. Product decides the interaction model — chat, proactive notifications, batch review, ambient automation. Intelligence fills those containers with good decisions and useful output. The better intelligence gets, the more ambitious the product can be.

Layer 1 — The Surface

UX / Product

What users see and interact with. The design, the interaction model, the trust signals, the feedback mechanisms. Product turns invisible intelligence into visible value. It decides how to present decisions, how to collect corrections, how to build trust over time.

Owns: User experience, interaction design, trust building, feedback collection, the "feel" of using Sauna.

Success looks like: Users trust the agent. They use it daily. They tell other people about it. They feel like it understands them. The product makes the intelligence accessible without dumbing it down.

Why this decomposition matters

Clear layers enable parallel progress. Infrastructure can scale without touching intelligence. Intelligence can iterate on prompts without touching capabilities. Product can redesign the experience without changing the reasoning underneath. Each layer has its own rate of change, its own team, its own metrics.

The alternative — a monolith where everything is tangled — means every change touches everything. A prompt improvement breaks a tool integration. A product redesign requires rethinking the reasoning. Nothing moves fast because everything is coupled.

The stack model isn't just architecture. It's an organizational principle. It tells you who owns what, where the boundaries are, and what contracts need to hold for the system to work. When something breaks, the stack tells you which layer failed and who needs to fix it.

The stack is a bet on clarity. In a fast-moving space where everything changes every quarter, the teams that move fastest are the ones that know exactly what they own and exactly what they can depend on. Four layers. Clear contracts. Parallel progress.
