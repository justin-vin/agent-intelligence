Evals &
Measurement

Evidence over vibes

Without measurement, every prompt change is a coin flip. You think it got better. Maybe it did for the case you tested. Maybe it broke three others you didn't check. Evals are the difference between engineering and guessing. They turn intelligence work from an art into a science — or at least a craft with feedback loops.

Why it's hard

Agent intelligence isn't a benchmark. There's no single score that captures "did it make a good decision." The outputs are natural language, the contexts are infinitely varied, and "good" is subjective — what's a perfect email for one person is wrong for another.

This doesn't mean measurement is impossible. It means we need to be thoughtful about what we measure and honest about what the measurements tell us. A bad metric measured precisely is worse than no metric at all — it gives false confidence.

What to measure

Decision Quality

Did it choose right?

Given the context and the task, did the agent make the decision a competent human would make? Not the creative decision, not the surprising decision — the right decision. Build golden sets: curated examples where you know what the right answer is, and score against them.

Task Completion

Did it finish the job?

Binary but important. Did the email get drafted? Did the meeting get scheduled? Did the research get synthesized? Partial completions count — track completion rate and identify where tasks stall.

Faithfulness

Did it stay true to the context?

Did it use the information provided or hallucinate? Did it follow the constraints or ignore them? Did it represent the user's preferences accurately or drift? Faithfulness to context is the most measurable aspect of intelligence.

User Trust Signals

Did they accept it?

The ultimate eval: did the user send the draft as-is, edit it heavily, or throw it away? Did they accept the recommendation or override it? Every user interaction is an implicit evaluation. Track the accept/edit/reject ratio.

Calibration

Does it know what it doesn't know?

When the agent says "I'm confident about this," is it right? When it says "I'm not sure," is that appropriate? Calibration — the alignment between stated confidence and actual accuracy — is one of the most important and underrated metrics.

The eval stack

Golden sets

Curated input/output pairs where you've defined what "good" looks like. Start small — 20-30 cases that cover your core scenarios. Expand as you discover edge cases. Every bug report is a potential golden set entry.

Regression suites

Before you change anything, run the suite. After you change something, run it again. Did the thing you fixed actually improve? Did anything else break? This is table stakes in software engineering. It should be table stakes in intelligence work too.

A/B comparison

Same input, two different prompts (or models, or context strategies). Which output is better? Not "which looks better to you right now" — which scores higher against your golden set? Which gets more user accepts? Controlled comparison beats gut feeling.

Traces & decision logs

Record the full reasoning path. What context was in the window? What decisions did the model make? Where did it branch? When something goes wrong, the trace tells you why. Without traces, debugging intelligence is archaeology.

Human-in-the-loop eval

User corrections are the highest-signal eval data. When someone edits a draft, the diff between original and edited version tells you exactly where the agent got it wrong. Collect these systematically. They're gold.

Continuous vs. point-in-time

Point-in-time evals are checkpoints. You run them before a release, after a major change, during a sprint review. They tell you where things stand right now.

Continuous evals run in production, silently. Every user interaction generates eval signal: accept rates, edit distances, task completion, error frequency. Over time, this paints a picture of intelligence quality that no test suite can match — because it's measured against real users doing real work.

You need both. Point-in-time for controlled comparison. Continuous for the ground truth. The gap between them tells you how well your test suite represents reality.

The measure of intelligence isn't what the system prompt says. It's whether the user trusted the output enough to act on it.

We're flying blind right now. That's the honest assessment. The first step isn't building a perfect eval framework — it's building any eval framework. Golden sets for our top five use cases. Regression tracking for prompt changes. Accept rate monitoring in production. Start there, iterate from evidence, and build the muscle of measuring before changing.
