Prompting Strategy

The craft of deterministic intent, probabilistic execution

A prompt is a program written in natural language. It has intent (what you want), constraints (what you don't want), and context (what the model needs to know). The difference between a good prompt and a bad one isn't cleverness — it's clarity. The model will try to do exactly what you ask. The hard part is asking for exactly what you mean.

The prompting spectrum

Not all prompting is the same activity. There's a spectrum from rigid instruction to open collaboration, and the right approach depends on what you're building.

Instruction

"Do exactly this, in this order, with these constraints."

High control, low flexibility. Best for defined workflows, data transformation, format enforcement. The agent is an executor.

Collaboration

"Here's the goal and context. Figure out the approach."

Low control, high flexibility. Best for research, exploration, creative work. The agent is a partner.

Most production prompting lives in the middle: guided autonomy. You set the boundaries, define the goal, provide the context — then let the model choose how to get there within those constraints. Think of it as hiring: you describe the role, not every task.

Seven principles

1. Teach, don't tell

The best system prompts describe a role and a way of thinking, not a list of rules for every situation. A prompt that says "you are a careful editor who values clarity over cleverness" will generalize better than one that enumerates 50 editing rules.

Rules are brittle. Mental models are flexible. Invest in shaping how the agent thinks, not just what it does.

2. Context is king

A mediocre prompt with excellent context will outperform a brilliant prompt with no context, every time. The model's reasoning is only as good as the information it has. Before optimizing your instructions, ask: does the model have everything it needs to make a good decision?

This is why context engineering matters more than prompt engineering. The prompt is the steering wheel. Context is the road.

3. Be specific about what "good" looks like

Vague instructions produce vague results. "Write a good email" is almost useless. "Write a 3-paragraph email to an investor, professional but warm, acknowledging their concern about burn rate and redirecting to our 70-month runway" gives the model something to work with.

The more concrete your success criteria, the better the output. Include examples when possible — they're worth more than paragraphs of instruction.

4. Separate concerns

System prompts, user prompts, and context serve different purposes. Don't mix them. The system prompt defines the agent's character and constraints. User messages carry the immediate task. Context provides the information needed to execute.

When these get tangled — behavioral instructions in user messages, context in system prompts, tasks in both — the model struggles to prioritize and contradictions emerge.

5. Design for the failure mode

Every prompt has a failure mode. The question isn't whether it will fail — it's whether the failure is graceful. A good prompt anticipates the edge cases: what should the agent do when it's unsure? When data is missing? When instructions conflict?

"If you're not confident, say so and explain why" is worth more than 10 lines of happy-path instructions.

6. Iterate with evidence

Prompt development is empirical, not theoretical. Write a prompt, test it against real cases, observe the failures, adjust. Keep a test suite of inputs and expected outputs. When you change something, run the suite. Did it get better? Did anything regress?

Intuition helps you write the first draft. Evidence helps you write the tenth.

7. Less is more (usually)

Long prompts aren't necessarily better prompts. Every additional instruction competes for attention in the context window. The model can follow 5 clear rules better than 50 overlapping ones. When in doubt, cut.

The exception: examples. Examples can be long. They're the highest-bandwidth way to communicate intent. One good example is worth a paragraph of explanation.

Common anti-patterns

The instruction novel

System prompts that try to cover every possible scenario with explicit rules. 10,000 tokens of instructions that the model can't possibly follow all at once. The fix: distill to principles, use examples for edge cases, and accept that some judgment calls are better left to the model.

Contradictory constraints

"Be concise but thorough. Be creative but stick to the facts. Be formal but friendly." Every contradiction forces the model to choose, and it might not choose the way you'd want. When you catch yourself writing "but," that's a signal to clarify your actual priority.

Negative instructions

"Don't use jargon. Don't be condescending. Don't start with 'I.'" Negatives are processed less reliably than positives. Instead of "don't use jargon," try "use plain language that a smart non-expert would understand." Tell the model what to do, not what to avoid.

Context starvation

Brilliant instructions with no context. "Write the perfect follow-up email" without including the original email, the relationship context, or the desired outcome. The model will produce something generic because that's all you gave it to work with.

One-shot optimization

Spending hours perfecting a single prompt in isolation. Prompts live in systems — they interact with context, tools, and other prompts. Optimize the system, not the prompt. A mediocre prompt in a well-designed system beats a perfect prompt in a broken one.

System prompts vs. user prompts

These are different tools. Confusing them is one of the most common mistakes.

The system prompt is the agent's identity and operating manual. It defines character, constraints, capabilities, and default behaviors. It should be stable — not changing per-turn. Think of it as the job description: who you are, how you work, what your boundaries are.

The user prompt is the task. It changes every turn. It says what to do right now, with what inputs, toward what output. It can reference the system prompt's constraints ("following the format guidelines") but shouldn't redefine them.

Context is the raw material. Documents, conversation history, tool outputs, user preferences. It goes alongside the user prompt, not inside the system prompt. The system prompt should reference that context exists, not contain it.

When all three are clean and separated, the model knows what it is, what it's doing, and what it's working with. When they're tangled, you get confused outputs and unpredictable behavior.

Prompting strategy isn't about tricks. It's not about "jailbreaks" or "magic words." It's about clear communication with a system that takes you literally. The models are getting better at understanding intent. But they'll always reward clarity, specificity, and good context. The best prompt engineers aren't the cleverest — they're the clearest thinkers.
